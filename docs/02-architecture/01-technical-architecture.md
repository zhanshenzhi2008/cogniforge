# CogniForge 技术架构设计文档

## 1. 技术架构概览

### 1.1 设计原则

| 原则 | 描述 |
|-----|------|
| **高性能** | AI推理延迟要求极高，核心路径采用Go实现 |
| **可扩展** | 微服务架构，支持水平扩展 |
| **多语言融合** | 根据服务特性选择最优技术栈 |
| **云原生** | 容器化部署，Kubernetes编排 |
| **可观测** | 全链路追踪、指标监控、日志聚合 |

### 1.2 技术栈选择理由

| 技术 | 选型理由 | 适用场景 |
|-----|---------|---------|
| **Go** | 高并发、低延迟、编译型无GC暂停 | API网关、模型网关、Agent引擎、工作流编排 |
| **Java (Spring Boot 3)** | 成熟稳定、丰富生态、企业级特性 | 用户中心、计费系统、审批流程 |
| **Python** | AI/ML事实标准、丰富库支持 | 模型微调、向量 embedding、RAG处理 |
| **React + TypeScript** | 组件化开发、强类型支持 | Web控制台前端 |
| **PostgreSQL** | ACID事务、JSON支持 | 核心业务数据存储 |
| **Redis** | 高速缓存、会话存储 | 缓存、限流、实时数据 |
| **Milvus/Qdrant** | 高效向量检索 | 知识库语义搜索 |
| **Kafka** | 高吞吐消息队列 | 异步任务、事件流 |

## 3. 核心服务设计

### 3.1 API 网关服务 (Go)

**职责**: 统一入口、认证鉴权、限流、路由、监控

```yaml
服务名称: gateway
语言: Go 1.22+
框架: Gin / Fiber
端口: 8080

核心功能:
  - JWT认证/验证
  - API密钥验证
  - 请求限流 (令牌桶)
  - 路由转发
  - 熔断器模式
  - 请求日志
  - 指标暴露 (Prometheus)

性能指标:
  - 单实例QPS: 50,000+
  - P99延迟: < 10ms
  - 内存占用: < 100MB
```

### 3.2 模型网关服务 (Go)

**职责**: 统一模型接入、负载均衡、故障转移

```yaml
服务名称: model
语言: Go 1.22+
框架: Gin + gRPC
端口: 8081

核心功能:
  - 多供应商适配 (OpenAI, Anthropic, Google, Cohere)
  - 模型版本管理
  - 负载均衡 (轮询/加权/最少连接)
  - 故障自动转移
  - 请求重试
  - 响应缓存
  - Token计数

支持的模型:
  - OpenAI: gpt-4o, gpt-4o-mini, o1, o1-mini
  - Anthropic: claude-4-opus, claude-4-sonnet, claude-4-haiku
  - Google: gemini-2.0-pro, gemini-2.0-flash
  - Cohere: command-r-plus, command-r
  - 开源: llama, qwen, deepseek, glm
  - 本地: ollama, vllm
```

### 3.3 Agent 引擎服务 (Go)

**职责**: Agent生命周期管理、推理调度、工具执行

```yaml
服务名称: agent
语言: Go 1.22+
框架: Gin + gRPC
端口: 8082

核心功能:
  - Agent CRUD
  - 推理模式配置 (ReAct/CoT/ToT)
  - 工具注册与发现
  - 短期记忆管理 (滑动窗口)
  - 长期记忆管理 (向量存储)
  - 护栏检查 (输入/输出过滤)
  - 人机协作 (审核点)
  - 执行追踪

推理流程:
  1. 接收用户请求
  2. 构建提示词 (系统提示 + 历史 + 当前输入)
  3. 执行护栏检查
  4. 调用模型推理
  5. 解析模型输出 (工具调用/直接回复)
  6. 如需工具调用，执行工具
  7. 重复步骤3-6直到完成
  8. 执行输出护栏
  9. 更新记忆，返回结果
```

### 3.4 工作流编排服务 (Go)

**职责**: 工作流设计、执行调度、状态管理

```yaml
服务名称: workflow
语言: Go 1.22+
框架: Gin + gRPC
端口: 8083

核心功能:
  - 可视化流程设计器后端
  - 流程版本管理
  - 节点执行引擎
  - 条件分支处理
  - 并行执行协调
  - 循环控制
  - 异常处理
  - 执行状态追踪

执行模型:
  - 同步执行: 简单流程
  - 异步执行: 复杂流程 (状态机)
  - 定时执行: Cron调度

节点类型支持:
  - LLM节点
  - Agent节点
  - HTTP请求节点
  - 条件分支节点
  - 并行分支节点
  - 循环节点
  - 向量检索节点
  - 数据转换节点
  - 等待节点
  - 人工审核节点
```

### 3.5 知识库服务 (Go + Python)

**职责**: 文档管理、向量化、语义检索

```yaml
服务名称: knowledge
语言: Go (API) + Python (处理)
框架: Gin + FastAPI
端口: 8084

核心功能:
  - 文档上传与管理
  - 多种格式支持 (PDF, DOCX, MD, TXT, HTML)
  - 智能分块 (递归/标题/句子)
  - 向量化处理 (Python)
  - 语义检索
  - 源引用返回
  - 增量更新

处理流程:
  1. 文档上传
  2. 格式检测与转换
  3. 文本提取
  4. 分块处理
  5. 向量化 (embedding模型)
  6. 存储到向量数据库
  7. 建立索引

检索流程:
  1. 接收查询
  2. 查询向量化
  3. 向量相似度搜索
  4. 重排序 (可选)
  5. 返回结果及原文引用
```

## 4. 服务间通信

### 4.1 通信模式

| 模式 | 技术 | 适用场景 |
|-----|------|---------|
| **同步调用** | gRPC | 实时响应、低延迟场景 |
| **异步消息** | Kafka | 耗时任务、事件驱动 |
| **服务发现** | Consul/etcd | 服务注册与发现 |

### 4.2 gRPC 服务定义

```protobuf
// 通用请求/响应
syntax = "proto3";

package cogniforge;

service ModelGateway {
  rpc ChatCompletion(ChatRequest) returns (ChatResponse);
  rpc StreamChat(ChatRequest) returns (stream StreamChunk);
  rpc Embeddings(EmbeddingRequest) returns (EmbeddingResponse);
}

service AgentEngine {
  rpc CreateAgent(AgentConfig) returns (Agent);
  rpc Chat(ChatRequest) returns (ChatResponse);
  rpc StreamChat(ChatRequest) returns (stream StreamChunk);
}

service WorkflowEngine {
  rpc CreateWorkflow(Workflow) returns (Workflow);
  rpc Execute(ExecuteRequest) returns (ExecuteResponse);
  rpc GetExecution(GetExecutionRequest) returns (Execution);
}
```

### 4.3 消息队列主题

```yaml
Topics:
  - ai.request: AI请求日志
  - ai.usage: 用量统计
  - ai.workflow.execute: 工作流执行
  - ai.agent.tool: Agent工具调用
  - system.notification: 系统通知
  - system.audit: 审计日志
```

## 5. 数据架构

### 5.1 存储分层

| 层级 | 存储类型 | 用途 |
|-----|---------|------|
| **热数据** | Redis | 缓存、会话、实时数据 |
| **温数据** | PostgreSQL | 业务数据、事务数据 |
| **冷数据** | S3/对象存储 | 文档、文件、日志归档 |
| **向量数据** | Milvus/Qdrant | 知识库向量 |
| **分析数据** | ClickHouse | 用量统计、成本分析 |

### 5.2 数据隔离

```yaml
隔离策略:
  - 租户隔离: 逻辑隔离 (租户ID)
  - 数据加密: 传输TLS、存储AES-256
  - 敏感数据: PII脱敏处理
  - 备份策略: 日增量 + 周全量
```

## 6. 部署架构

### 6.1 Kubernetes 部署

```yaml
部署配置:
  - 集群: 3可用区，每区3节点
  - GPU: 按需弹性池
  - 副本数: 最小2，最大50
  - 自动扩缩容: HPA (CPU>70%, 内存>80%)
  - 健康检查: Liveness + Readiness

环境:
  - 开发: K3s (单节点)
  - 测试: K8s (3节点)
  - 生产: K8s (多可用区)
```

### 6.2 高可用配置

```yaml
高可用:
  - API网关: 3副本 + LoadBalancer
  - 核心服务: 3副本 + HPA
  - 数据库: 主从 + 自动故障转移
  - 缓存: Redis Cluster (6节点)
  - 消息队列: Kafka (3 broker, 3副本)
  - 向量库: Milvus Cluster (3节点)
```

## 7. 安全架构

### 7.1 认证授权

```yaml
认证方式:
  - API密钥: 用于程序调用
  - JWT Token: 用于用户会话
  - OAuth2/OIDC: 用于SSO登录

授权模型: RBAC
  - 超级管理员
  - 组织管理员
  - 开发者
  - 分析师
  - 只读用户
```

### 7.2 安全防护

```yaml
防护措施:
  - WAF: DDoS防护、SQL注入、XSS防护
  - 限流: 每IP/每用户/每API
  - 审计: 完整操作日志
  - 密钥管理: HashiCorp Vault
  - 敏感数据: 自动脱敏
```

## 8. 技术选型总结

### 8.1 服务技术矩阵

| 服务 | 语言 | 框架 | 原因 |
|-----|------|------|------|
| gateway | Go | Gin | 高性能、低延迟 |
| model | Go | Gin+gRPC | 高并发、AI请求处理 |
| agent | Go | Gin+gRPC | 实时推理、低延迟 |
| workflow | Go | Gin+gRPC | 高并发、执行调度 |
| knowledge | Go+Python | Gin+FastAPI | Go处理API, Python处理ML |
| user | Java | Spring Boot | 成熟稳定、事务处理 |
| billing | Java | Spring Boot | 业务复杂、需要事务 |
| monitor | Go | Gin | 高性能指标处理 |
| ml | Python | FastAPI | AI/ML事实标准 |

### 8.2 核心原则

> **"Go for serving, Python for training"**
>
> - 实时推理路径全部使用Go
> - AI/ML处理使用Python
> - 企业级业务使用Java
> - 前端使用React+TypeScript
